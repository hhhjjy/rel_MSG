====================
一、总体目标
====================
我要做的是一个渐进式实验路线，而不是一次性推翻 AoMSG。请始终以“最小改动、可回退、可比较、可做消融”为原则推进。

实验路线分四步：

Step 1. 基础 AoMSG 模型
- 保持现有 AoMSG 训练与评估逻辑不变，作为 baseline。
- 不要为了新方法去破坏这个版本的可运行性。

Step 2. 在 query 后增加类似 VGGT 的交替注意力机制，再输入 transformer decoder
- 尽量保持 query 的来源不变。
- 目标是验证 alternating attention + decoder 是否带来收益。
- 这一步不要引入 learnable query，不要同时引入场景图预测头，避免变量混杂。

Step 3. 将原本 query 替换为可学习向量，由 learnable queries 去查询经过 VGGT 后的 bbox 特征
- 目标是验证 learnable queries 是否优于原始 query 方案。
- 保持 Step 2 中的交替注意力与 decoder 结构尽量不变。
- 这一阶段仍然优先复用已有训练与评估框架，尽量不要一次性重写 evaluator。

Step 4. 加入显式场景图预测模块
- 目标是让模型显式输出 scene graph 相关预测，而不是完全依赖旧的隐式后处理。
- 请把这一步拆成两个子阶段：
  Step 4a. 先加 graph prediction heads，但尽量复用当前 evaluator 所需输出格式；
  Step 4b. 再逐步把训练目标与推理组织切换到更完整的 scene graph formulation。
- 不要一开始就把整个评估链条全部推翻。

四、关于评估与兼容性的特别要求
====================
请牢记：现有评估主线是稳定资产，尽量复用。

- 预测的 detections

- 后续构造/比较 p-o、graph_iou 等

那么请优先采用“适配输出”的方式兼容，而不是立刻重写 evaluator。

换句话说：
- 如果新模型输出的是 query features / node features / edge logits，
- 请优先设计一个 adapter / mapper，把这些输出整理成当前 evaluator 可消费的格式；
- 只有在确实无法表达时，才建议改 evaluator。

请始终优先考虑以下顺序：
1. 复用 evaluator
2. 新增 adapter / mapper
3. 最后才考虑重写 evaluator
